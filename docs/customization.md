# Customization Guide

## Manifest Tuning

The `autocode.manifest.json` is generated by `/autocode-bootstrap`, but you can edit it manually to tune behavior.

### Coverage Targets

Adjust how aggressive coverage goals are:

```json
{
  "coverage": {
    "targets": {
      "statements": 80,
      "branches": 70,
      "functions": 80,
      "lines": 80
    }
  }
}
```

Lower targets = factory stops sooner. Higher targets = more cycles but potentially harder-to-cover code.

### Coverage Gap Priority

Reorder which files get tackled first:

```json
{
  "coverage": {
    "gaps": [
      { "file": "src/critical-module.ts", "coverage": 5, "priority": 1 },
      { "file": "src/less-important.ts", "coverage": 10, "priority": 99 }
    ]
  }
}
```

Priority 1 is tackled first. Set high priority numbers to deprioritize files.

### Guardrails

Control what the factory can and cannot touch:

```json
{
  "guardrails": {
    "immutable_patterns": [
      "*.config.*",
      "*.env*",
      ".github/**",
      "autocode.manifest.json",
      "src/legacy/**"
    ],
    "max_files_per_pr": 3,
    "max_lines_changed": 100
  }
}
```

- **immutable_patterns**: Glob patterns for files agents must never modify. Add directories or file patterns you want protected.
- **max_files_per_pr**: Smaller = easier to review, but more PRs. Default: 5.
- **max_lines_changed**: Smaller = safer changes, but may limit what can be done. Default: 200.

### Time Budgets

Control how long each agent gets:

```json
{
  "time_budgets": {
    "scout_seconds": 120,
    "architect_seconds": 120,
    "builder_seconds": 300,
    "tester_seconds": 300,
    "reviewer_seconds": 180,
    "cycle_max_seconds": 1200
  }
}
```

Shorter budgets = faster cycles but agents may not finish complex tasks. Longer budgets = higher quality but higher cost.

### Difficulty

Control the starting difficulty and progression:

```json
{
  "difficulty": {
    "current_level": 1,
    "advance_when": "3 consecutive successful cycles at current level"
  }
}
```

- Set `current_level` to 3+ if your project already has good pure-function coverage
- The factory will auto-advance after 3 consecutive successes

### Model Routing

Control which AI models each agent uses:

```json
{
  "model_routing": {
    "scout": "haiku",
    "architect": "sonnet",
    "builder": "opus",
    "tester": "sonnet",
    "reviewer": "opus"
  }
}
```

- Use "haiku" for cost-efficient agents (Scout, simple Tester work)
- Use "sonnet" for mid-complexity agents (Architect, Tester)
- Use "opus" for high-judgment agents (Builder, Reviewer)

**Cost tradeoff**: Using Haiku everywhere is ~10x cheaper but produces lower-quality implementations. Using Opus everywhere produces better results but costs significantly more per cycle.

## Agent Customization

Agent definitions live in `agents/*.md`. You can modify them to:

### Add Project-Specific Context

Add a section to an agent file with project-specific instructions:

```markdown
## Project-Specific Notes

- This project uses a custom test helper at `src/test/helpers.ts` — always import `createTestContext` from there
- Database tests must use the `withTestDb` wrapper
- All async functions should be tested with both success and error paths
```

### Change Review Criteria

Edit `agents/reviewer.md` to add or remove review criteria. For example, add:

```markdown
### Performance
- No N+1 queries introduced
- No synchronous file I/O in request handlers
```

### Adjust Builder Patterns

Edit `agents/builder.md` to enforce specific patterns:

```markdown
## Additional Rules
- Always use `vi.mock()` for module mocking, never `jest.mock()`
- Test files must be co-located with source files (not in a separate `__tests__` directory)
- Use `faker` for generating test data
```

## Memory Management

### Clearing Memory

To reset the factory's memory (start fresh):

```bash
rm -rf .autocode/memory/
```

Next run will recreate the memory directory.

### Seeding Lessons

Pre-populate `.autocode/memory/lessons.md` with known patterns:

```markdown
# Lessons

## Testing Patterns
- Use `vi.spyOn` instead of `vi.fn` when you need to preserve the original implementation
- Always mock `Date.now()` in time-dependent tests

## Known Issues
- The `auth` module has circular dependencies — mock at the module level, not the function level
- `stripe.ts` requires a test API key in `.env.test`
```

### Marking Files as "Don't Retry"

Add entries to `.autocode/memory/failures.md` to prevent the factory from attempting specific files:

```markdown
## src/legacy/old-parser.ts — PERMANENT SKIP
- Reason: This file is scheduled for deletion in Q2
- Do not attempt coverage improvements
```

## Running Against Multiple Repos

AutoCode manifests are per-repo. To run against multiple projects:

1. Run `/autocode-bootstrap` in each repo
2. Run `/autocode` in each repo (in separate terminal sessions)

Each repo maintains its own manifest, memory, and worktrees.
